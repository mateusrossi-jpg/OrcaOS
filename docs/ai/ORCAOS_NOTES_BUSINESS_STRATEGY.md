# OrçaOS — decisões vindas das anotações manuais

Este documento registra decisões estratégicas capturadas das anotações manuais do Mateus para orientar o desenvolvimento do OrçaOS.

## Direção principal

O OrçaOS deve ser tratado como possível fonte de receita e produto comercial, não apenas como experimento.

Objetivo inicial:

- lançar rápido uma versão utilizável;
- validar com uso real;
- começar a gerar renda;
- manter base técnica evolutiva;
- priorizar estética, tipografia, usabilidade e experiência agradável;
- deixar projetos antigos em segundo plano até o app começar a mostrar potencial real de retorno.

## Modelo de receita pensado

Ideias registradas:

- primeira compra/licença;
- recorrência ou licença de 12 meses;
- possibilidade de plano anual com acesso completo;
- cálculos Pro atuais e futuros incluídos conforme política do plano;
- valor estudado como faixa inicial aproximada, ainda a validar com mercado;
- considerar aquisição/renovação recorrente para sustentar desenvolvimento.

A decisão comercial final ainda deve ser validada com usuários reais.

## Prioridade de lançamento

A prioridade é lançar a versão inicial o mais rápido possível para validação real.

Critérios:

1. app precisa ser estável;
2. app precisa ter uso prático claro;
3. app precisa parecer profissional;
4. app precisa justificar seus cálculos;
5. app precisa orientar o usuário sobre o próximo passo;
6. app precisa ser bom o suficiente para gerar confiança.

## Estética e experiência

A estética não é detalhe secundário. Ela é parte do valor percebido.

Diretrizes:

- aparência profissional;
- tipografia mais agradável;
- visual limpo;
- experiência satisfatória;
- usabilidade prazerosa;
- nada grosseiro ou improvisado;
- design deve passar confiança.

## Cálculos com justificativa

O OrçaOS deve evitar mostrar apenas um número final.

Cada cálculo deve evoluir para um modelo com:

- resultado principal;
- justificativa curta;
- fórmula/critério aplicado quando fizer sentido;
- orientação prática abaixo;
- alerta quando houver dependência de outros cálculos;
- sugestão do próximo cálculo necessário.

Exemplo de fluxo:

1. calcular corrente;
2. calcular queda de tensão;
3. aplicar fator de correção;
4. revisar seção/cabo/disjuntor;
5. apresentar orientação final.

O objetivo é acumular cálculos relacionados até obter um resultado final mais confiável e uma orientação prática.

## Resultado final orientado

Após cálculos relacionados, o app deve conseguir apresentar algo como:

- valor calculado;
- escolha recomendada;
- justificativa;
- cuidados de instalação;
- próximos pontos a verificar;
- aviso de validação profissional/norma quando necessário.

## Possível divisão em dois aplicativos

Ideia registrada para avaliação futura:

- **OrçaOS Empresa**: versão completa para profissional, técnico, eletricista, prestador de serviço e gestão;
- **OrçaOS Cliente**: versão mais simples para cliente final acompanhar orçamento, proposta, diagnóstico, status de OS e talvez aprovar serviços.

As duas versões poderiam conversar entre si no futuro.

Estratégia recomendada por enquanto:

- manter um único código-base;
- separar perfis/experiências por modo de uso;
- só dividir em apps separados depois que o fluxo principal estiver validado.

## Regra de foco

Não dispersar em muitos projetos antes do OrçaOS ter uma versão publicável e validada.

Prioridade de curto prazo:

1. estabilizar app;
2. fortalecer orçamento guiado;
3. fortalecer catálogo/fornecedor/estoque;
4. melhorar cálculos com justificativa;
5. melhorar proposta/orçamento;
6. melhorar visual;
7. lançar para validação real.
